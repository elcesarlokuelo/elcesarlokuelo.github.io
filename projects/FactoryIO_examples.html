Factory IO Examples

Factory IO · PLC · Ladder · Codesys · TIA Portal

This page contains a collection of solved examples for the Factory IO software, which is a 3D simulation platform for industrial automation. The examples cover various scenarios and challenges, and provides solutions using different PLC programming environments such as Codesys and TIA Portal and different communication protocols like Modbus, Profinet or OPC-UA.

All factory IO files and project resources are available in the repository: https://github.com/elcesarlokuelo/elcesarlokuelo.github.io/tree/main/assets/Downloads/Solved_factoryIO_examples

Each example will include a description of the problem, the implemented solution in the chosen PLC code language, and the corresponding software and communication protocol used. For all the examples there will be a video demonstration of the solution in action within the Factory IO simulation environment, and a link for downloading the project files.

Assembler

The objective in this scene is to assemble parts made of lids and bases using a two-axis pick and place. For this example I used CODESYS and Modbus TCP to communicate with Factory IO. The PLC program is written in Ladder logic, using a finite state machine to control the pick and place operation.

First we define the necessary I/O as global variables, looking at the Factory IO driver configuration. We also have to link the variables to the corresponding Modbus addresses in the PLC configuration, in the holding registers, coils or discrete inputs.

The control logic is implemented as a state machine using Set/Reset coils, where each state (E0 to E13) represents a step in the assembly sequence. Pressing Start initiates two parallel branches: one state machine for the lids conveyor and another for the bases conveyor, since both parts are handled independently until they reach the clamping station. Each state is activated by a condition (a sensor detecting a part, a clamp closing, an axis finishing its move) and immediately resets the previous state, ensuring only one state is active at a time per branch. Timers are used where there is no direct sensor feedback, such as during the Z-axis movement or the pick and place cycle. The sequence finishes by counting the finished pieces with a falling-edge triggered counter and returning the system to its initial states, ready for a new cycle.

Assembler (Analog)

This example is very similar to the previous one, but in this case we use analog inputs and outputs to control the pick and place axes and positions. I used Codesys and Modbus TCP to communicate with Factory IO. The PLC program is written in Ladder logic, using a finite state machine to control the pick and place operation.

For this example i used the CADe_SIMU software to create a grafcet diagram of the process, making it easier to then translate it into the PLC ladder program.

We also have to define the necessary I/O as global variables, looking at the Factory IO driver. Then we have to link the variables to the corresponding Modbus addresses in the PLC configuration, in the holding registers, coils or discrete inputs.

I use 2 different functions to implement the toggle button logic for the stop push button and the counter for the total finished pieces.

The program implements a finite state machine with three parallel branches: one tracks the lid sensor on conveyor 1, another tracks the base sensor on conveyor 2, and the third controls the robotic arm's movement (X/Z positioning, grabbing and releasing parts using position comparisons). State E0 acts as the master reset, restarting all three branches when Start is pressed. At the end of the program, the states are mapped to the physical outputs: conveyors, stop blades, and other station actuators.

Buffer Station

The objective of this example is to buffer and separate up to five boxes, using a series of conveyors and sensors. For this example I used CODESYS and OPC-UA, using the simulated PLC as an OPC-UA server to communicate with Factory IO acting as a client. The PLC program is written in Ladder logic. Additionally I created a simple HMI to visualize the status of the system and control the operation when the manual mode is selected.

Firstly we create the global variables and a symbol configuration object to define the I/Os and expose them via OPC-UA. In this case we have 8 digital inputs and a REAL input for the potentiometer used for the conveyors velocity. We also define 7 outputs for the analog conveyors, displays, button lights and the stop blade.

I used the PLC_PRG POU to organize the program modes, and the general logic of the emergency, start, reset and stop buttons. In the HMI I created there are the same controls as in the panel in Factory IO, and also the manual controls of the outputs. The idea is that the operator can control the process manually from the HMI, or let the PLC control the process automatically using the "physical" control panel. The mode can only be changed via the rotary switch on the physical panel, avoiding control source conflicts between the HMI and the simulation.

The manual mode consist only in the direct control of the outputs and the conveyor velocity. Also i can duplicate the start, stop and reset buttons in the HMI because they are momentary, and cant conflict with the physical buttons in the panel.

The automatic mode does batches of three boxes. The operator can start, reset or stop the process in the physical panel or the HMI, but can't adjust the conveyor velocity or control the outputs directly from the HMI.

Filling Tank (Timers)

The objective of this example is to fill and empty a tank using timers. For that we have 2 valves, one for filling and another for emptying. We also have 2 buttons and a display in a control panel. For this example I used CODESYS and Modbus TCP to communicate the simulated PLC with Factory IO. The PLC program is written in Ladder logic.

We must define the mentioned I/O as global variables. In this case we have 2 digital inputs for the fill and discharge buttons, 2 digital outputs for the valves, 2 digital outputs for the button lights, and an integer output for the display. I also use an auxiliary variable for the Int to Word conversion, since the display only accepts Word values.

Then we create a Ladder logic program that turns on the filling valve for 9 seconds and discharges for 9 seconds. The button lights act as a latching variable, and the display shows the remaining time for each operation. Finally, we map the I/O variables to the corresponding addresses in the Modbus TCP server and run the simulation in Factory IO.

From A to B

The objective of this example is to transport a box until it reaches a sensor, using a conveyor belt and a retroreflective sensor. For this example I used CODESYS and Modbus TCP to communicate the simulated PLC with Factory IO. The PLC program is written in Ladder logic. First, we must define the necessary I/O as global variables. In this case we have a digital output for the conveyor belt and a digital input for the sensor.

Then we create a simple Ladder logic program that turns on the conveyor belt when the sensor is not activated, and turns it off when it is, keeping in mind that the sensor is NC. Finally, we map the I/O variables to the corresponding addresses in the Modbus TCP server and run the simulation in Factory IO.

Level Control

The objective of this example is to control the liquid level of a tank using a Proportional-Integral-Derivative (PID) controller. I used CODESYS as a SoftPLC and Modbus TCP to communicate with Factory IO. The program combines Ladder Diagram (LD) for the control logic (Start/Stop/Reset and interlocking) with Structured Text (ST) for scaling and PID configuration.

A latching circuit handles the Start/Stop logic: pressing Start energizes the "Marcha" coil, which keeps itself active through its own contact until Stop is pressed. This "Marcha" bit enables the rest of the process, including the PID block and the fill valve.

Two ST routines handle the scaling: the setpoint coming from a potentiometer and the process value coming from the level sensor are both rescaled from the raw analog range (0–1000) to the 0–300 mm range of the tank. Once the system is running, the PID output is converted with TO_WORD and written to the fill valve through a MOVE instruction.

Communication between CODESYS and Factory IO is done through Modbus TCP, mapping Holding Registers for the level sensor and setpoint, Input Registers for the valve commands and PID values, and discrete Coils/Inputs for the Start/Stop/Reset buttons and their indicator lights.
